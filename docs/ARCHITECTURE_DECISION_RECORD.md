# Architecture Decision Record

## ADR-001: Star schema over normalized Gold layer
**Decision:** Model the Gold layer as a star schema (4 dims, 3 facts) rather
than a normalized relational model.
**Rationale:** Optimizes for Power BI query performance and DAX simplicity.
Facts store only surrogate keys and measures; all descriptive attributes
live in dimensions.

## ADR-002: SCD Type 2 only on dim_employee
**Decision:** Only `dim_employee` uses SCD Type 2 versioning.
`dim_department`, `dim_date`, `dim_job_grade` are Type 1 (overwrite).
**Rationale:** Department/job grade/manager changes for an employee are the
only attribute changes that matter for historical accuracy at the volume
and use cases this project targets. Departments and grades themselves
changing structurally is rare enough to not warrant Type 2 overhead here.

## ADR-003: Surrogate keys are BIGINT, not INT
**Decision:** All IDENTITY surrogate keys use BIGINT.
**Rationale:** Fabric Warehouse requires IDENTITY columns to be BIGINT: INT
is rejected outright. Adopted as the standard for all surrogate keys for
consistency, even on dimensions unlikely to ever approach INT's range.

## ADR-004: As-of joins for all SCD2 fact loads
**Decision:** Fact table loads join to `dim_employee` using
`fact_date BETWEEN effective_start_date AND effective_end_date`, not a
plain equi-join on employee_id.
**Rationale:** Without this, a payroll or attendance record could join to
whichever dim_employee row happens to currently be `is_current = 1`, which
is wrong for historical fact data recorded before a later attribute change.

## ADR-005: Headcount snapshot as a generated time series, not a raw load
**Decision:** `fact_headcount_snapshot` is not a 1:1 load from a source
file -- it's generated by cross-joining every employee against 12
month-end dates and filtering to dates on/after each employee's hire date.
**Rationale:** The source data (`Employee_Master.csv`) only has
`HireDate`/`TerminationDate`, not month-by-month history. Recomputing
active/new-hire/termination status per snapshot date from those two
columns is the standard pattern for headcount trend reporting, and trades
storage for query-time simplicity versus deriving it from SCD2 history at
query time.

## ADR-006: Substituted metrics where no real data source exists
**Decision:** Several task-sheet-specified metrics were substituted with
the closest defensible alternative rather than fabricated:
- "Salary Variance % vs Budget" → variance vs. grade band midpoint
  (no budget data source exists anywhere in the provided files)
- "Open Requisitions" KPI → replaced with Avg Tenure (Years)
  (no recruiting/requisition data source exists)
- Voluntary/Involuntary/Retirement/Contract End funnel → built once
  `exit_reason` and a derived `attrition_category` were added to
  `dim_employee` (originally missing from the Gold schema)
**Rationale:** Honest substitution with clear documentation is preferable
to either fabricating data or silently omitting the requirement.

## ADR-007: Map visual replaced where licensing didn't allow it (initially)
**Decision:** Where Fabric Map/Filled Map visuals were blocked by tenant
and Power BI Desktop security settings, a sorted bar chart was prepared as
a fallback before both settings were successfully enabled.
**Rationale:** Region-level geographic data (e.g. "Asia Pacific") doesn't
have precise map coordinates in the first place -- a ranked bar chart is
arguably clearer at this level of granularity regardless of licensing.

## ADR-008: Q&A visual descoped, Copilot unavailable
**Decision:** Natural-language query (Task 6.1/6.2 in Scenario 5) was not
built as originally specified.
**Rationale:** Microsoft deprecated Q&A visual creation ahead of full
retirement in December 2026, with Power BI Copilot as the designated
replacement. Copilot visuals require an F64+ Fabric capacity tier, not
available on a trial/guest tenant. This is a genuine platform/licensing
constraint, not a data or implementation gap.
